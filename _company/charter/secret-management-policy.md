# 秘密情報管理方針(Secret Management Policy)

## 位置づけ
本ポリシーは、AI Company全体に適用される全社共通の憲章・方針(`_company/charter/`)であり、特定チャネル・特定Divisionに専属するSOPではない。将来API・MCP接続を実際に行う前に整備すべき、秘密情報の分類・保存場所・発行原則・ローテーション/失効・漏えい時対応・ログ衛生・認証情報台帳の枠組みを定義する。**本ポリシーは秘密情報の実値を一切保持しない。値そのものの管理はローカル環境・承認済みパスワード管理サービス・将来のCI/CDシークレットストアで行い、本ポリシーは「どう扱うか」というルールのみを定める。**

`_shared/sop/`配下のSOPとしては新設せず、`_company/charter/ai-usage-policy.md`・`outbound-action-policy.md`と同様に全社Charter/Policyの1つとして扱う(そのため`_shared/sop/sop-index.md`は今回変更しない)。

## 1. 秘密情報の分類

### 秘密情報(絶対に公開・共有しない)
APIキー、アクセストークン、リフレッシュトークン、クライアントシークレット、パスワード、秘密鍵、Cookie、セッション情報、Webhook署名秘密鍵、サービスアカウント秘密情報、復旧コード、二要素認証コード。

### 通常は秘密ではないが管理が必要な識別情報
アプリID、アカウントID、プロパティID、チャンネルID、公開URL、ユーザー名、公開鍵、秘密値を含まないWebhook URL。**ただし、これらの識別情報も、他の情報と組み合わせることで悪用され得るため、不用意に公開しない方針とする。**

## 2. 許可する保存場所

**ローカル開発時:**
- CEO管理PCのローカル`.env`ファイル(Git管理対象外)
- OSの環境変数
- CEOが承認したパスワード管理サービス

**将来CI/CDを導入した場合:**
- GitHub Secrets等の承認済み秘密情報ストア

**現時点の状態:** `selected_password_manager: not_decided`(承認済みパスワード管理サービスは未確定。特定製品を本ポリシーが勝手に指定することはしない。CEOが選定・承認した時点で本欄を更新する)

## 3. 禁止する保存場所
Git管理対象ファイル全般(Markdown・YAML・README・Agent定義・Knowledge・投稿原稿・calendar.md)、Gitコミットメッセージ、Issue、Pull Request本文、チャット本文、スクリーンショット、画像ファイル、ログ、CEO承認記録、公開用テンプレート。

## 4. `.env.example`の原則
- 実際のAPIキー・アクセストークン・パスワード・Cookie・秘密鍵・クライアントシークレット・実在するユーザーID・個人メールアドレス・電話番号・本番URLに含まれる秘密パラメータ・実在するアカウント識別子・秘密情報に見えるランダム文字列・過去に使用した認証情報を一切記載しない
- 値は空欄にするか、明らかに非秘密の説明コメントのみとする(例: `OPENAI_API_KEY=`)
- 実在する値に見えるサンプル文字列は使用しない
- 環境変数名は、`mcp/api-registry.yaml`・`mcp/servers.yaml`に登録済みの候補サービスのうち、**公式ドキュメントで名称を確認できたものだけ**を記載する。確認できない場合は推測で追加せず、確認不能として報告する

## 5. Claude Code・GPT・Fable 5への入力禁止
以下の秘密情報を、Claude Code・GPT・Fable 5その他のAIチャットへ**直接貼り付けない**。
- APIキー、アクセストークン、パスワード、Cookie、二要素認証コード、秘密鍵、復旧コード、認証済みURL、サービスアカウントJSONの内容

AIへ確認させてよいのは、変数名・設定項目名・エラーコード・秘密値を除去したエラーメッセージ・秘密部分を完全に伏せた設定構造のみとする。**「一部だけ伏せる」では不十分な場合があるため、秘密値全体を除去する方針とする。**

## 6. 認証情報の発行原則
将来API接続を行う場合、以下を原則とする。
- 必要なサービスだけ発行する
- 必要最小限の権限とする
- 可能なら読み取り専用から開始する
- 本番とテストを分離する
- 個人用トークンの使い回しを避ける
- 用途ごとに識別できる名前を付ける
- 発行日と所有者を、秘密値とは別に記録する
- 有効期限を確認する
- 不要になったら即時失効する
- 退職・端末紛失・権限変更時は交換する
- 外部書き込み権限は、`_company/charter/outbound-action-policy.md`に基づきCEO承認後にのみ付与する

## 7. ローテーション・失効ルール
以下の場合、認証情報を失効・交換する。
漏えいまたは漏えい疑い、Gitへの誤コミット、AIチャットへの誤貼り付け、スクリーンショットへの映り込み、PC紛失または盗難、関係者の権限変更、外部サービス側の警告、不審なAPI利用、用途終了、有効期限到来、権限を過剰に付与したことが判明した場合。

**失効・交換の順序:**
1. 外部サービス側で失効または交換する
2. 不正利用の有無を確認する
3. ローカル環境の値を更新する
4. 関係するサービス・処理を再確認する
5. 監査記録へ事実を記録する
6. 必要に応じてGit履歴対応を検討する

**漏えい時は、ファイルを削除するだけでは不十分である。** 削除しても値自体はGit履歴やコピー先に残り得るため、必ず外部サービス側での失効・交換を最優先で行う。

## 8. 誤コミット・漏えい時の対応(フェイルクローズ手順)
1. 関係する外部処理を停止する
2. 秘密情報を即時失効または交換する
3. CEOへ報告する
4. 影響範囲を確認する
5. 不正利用履歴を確認する
6. リポジトリ内の追跡状態を確認する
7. 新しい秘密情報を安全な保存先へ設定する
8. 再発防止策を記録する
9. 必要な場合のみGit履歴から除去する

**Git履歴の書き換えは破壊的操作であるため、CEOの明示的承認なしに実行しない。**

秘密情報らしき値が発見された場合、値を一切表示せず、以下のみを報告する: 該当ファイルパス、秘密情報の種類の推定、Git追跡状態、必要な緊急対応。

## 9. ログへの秘密情報混入防止
以下をログへ出力しない。
環境変数の全一覧、Authorizationヘッダー、Cookie、リクエスト本文内の秘密情報、アクセストークン付きURL、サービスアカウントJSON、秘密鍵、パスワード、Webhook署名秘密鍵。

将来デバッグ時も、秘密部分を完全に削除または安全にマスクする。マスク例を記載する場合も、実在する秘密値に見える文字列は使用しない。

## 10. 認証情報台帳(項目定義のみ。実レコードは今回作成しない)
秘密値そのものを保存しない前提で、将来の認証情報管理台帳に最低限含める項目。

```yaml
# サンプル(項目定義のみ。実在する認証情報のレコードではない)
sample_only: true
do_not_import: true
credential_id: null           # 例: "cred-line-channel-access-token-01"
service: null                 # 例: "line-api"
purpose: null
owner: null
environment: null             # production / test / development等
permission_scope: null
issued_at: null
expires_at: null
last_rotated_at: null
storage_location_type: null   # local-env / os-env / password-manager / github-secrets等
status: null                  # active / expired / revoked
revocation_reference: null    # 失効・交換を行った場合の参照(実値は含めない)
notes: null
```
台帳には実際のキー・トークン・パスワードを絶対に記載しない。
